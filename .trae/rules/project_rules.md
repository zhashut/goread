1. 注释要用中文且不要有 ai 味，生成的注释只描述代码作用，不要描述你做了什么，修改了什么，只描述代码作用！
2. 让生成 commit 信息的时候，要总结对话的内容，生成一条总结性的信息，格式举例如下，下面根据对话选择对应的格式：
    fix({具体模块}): 修复{具体模块}的问题
    feat({具体模块}): 新增{具体模块}的功能
    perf({具体模块}): 优化{具体模块}的性能
    refactor({具体模块}): 重构{具体模块}的代码
    chore({具体模块}): 其他{具体模块}的修改
3. 我需要特别提醒，你的优化和修复，绝对不能影响到其他书籍格式的功能和逻辑，每个书籍格式的逻辑封装到它自己的实现中去，绝对不要影响其他格式的逻辑，对于共用的逻辑修改要谨慎。
4. 在编写代码时，请优先使用抽象接口而不是具体类。核心逻辑应该依赖于接口定义，而不是具体的实现细节。
5. 生成代码时请遵循DRY原则（Don't Repeat Yourself），将重复的逻辑提取到公共函数或基类中
6. goread 项目是 Tauri V2(React/Rust) 架构，主要做的是移动端(android 和 ios) 的电子书阅读器
7. 如果修改了前端文件最后要进行 npm run build 检查问题
8. 如果修改了后端文件最后要进行 cargo build 检查问题
9. 本项目前端日志埋点要使用 src\services\index.ts 中的 log or logError 函数，后端使用 println! 打印日志。
10. 逻辑能抽成 Hook 的抽成 Hook，通用逻辑抽到 `src/utils`，一个函数不要臃肿，可以拆成多个小函数，但是要保证职责单一。
